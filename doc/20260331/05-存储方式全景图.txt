================================================================================
二、文档数据库
================================================================================

代表: MongoDB, CouchDB, Azure Cosmos DB

核心优势:

    ✓ Schema-free
      字段可随时增删，灵活应对变化

    ✓ 嵌套文档
      一个文档存完整对象，不需要 JOIN

    ✓ JSON 原生
      与现代 Web/AI 应用天然契合

    ✓ 水平扩展
      分片集群，大数据友好

本质局限:

    ✗ JOIN 能力弱
      不擅长跨文档关联

    ✗ 事务弱于关系库
      多文档事务成本高

    ✗ 精确查询不如 SQL 灵活

适合 Agent 记忆吗:

    适合: 原始对话（结构不固定）、配置文件、用户画像
    不适合: 需要复杂查询、需要严格事务的数据

对比 SQLite:

    MongoDB 更灵活，但 SQLite 更轻量、查询更强
    单机小规模，SQLite 更合适；大规模分布式，MongoDB 有优势

================================================================================
三、键值数据库
================================================================================

代表: Redis, DynamoDB, LevelDB

核心优势:

    ✓ 极速读写
      微秒级响应，内存级性能

    ✓ 简单模型
      一个 key 对应一个 value，无复杂查询

    ✓ 分布式扩展
      天然支持集群

    ✓ 多数据类型
      String, List, Set, Hash, Sorted Set

本质局限:

    ✗ 无复杂查询
      只能按 key 找，不能按 value 内容找

    ✗ 无事务
      单 key 操作原子，多 key 无保证

    ✗ 无结构化
      value 是黑盒，无法解析内部字段

适合 Agent 记忆吗:

    适合: 缓存层、临时状态、会话 token、高频访问的热数据
    不适合: 长期存储、需要检索的内容

典型用法:

    Redis 做热数据缓存（最近 100 条对话），SQLite 做长期存储

================================================================================
四、搜索引擎
================================================================================

代表: Elasticsearch, Meilisearch, Sphinx

核心优势:

    ✓ 全文搜索
      倒排索引，毫秒级搜索海量文本

    ✓ 复合查询
      全文 + 结构化条件混合

    ✓ 聚合分析
      统计、分组、计数

    ✓ 分布式
      TB 级数据也能快速搜索

本质局限:

    ✗ 不擅长精确查询
      不是关系库，JOIN 弱

    ✗ 写入成本高
      索引构建开销大

    ✗ 存储成本高
      索引比原始数据大

    ✗ 事务弱
      不适合频繁更新

适合 Agent 记忆吗:

    适合: 大量文本的全文搜索、日志分析、对话检索
    不适合: 频繁更新的数据、需要事务的数据、小规模场景

对比 SQLite FTS5:

    Elasticsearch 更强大，但 SQLite FTS5 更轻量
    数据量 <100 万条，SQLite 够用；>1000 万条，考虑 ES

================================================================================
五、列式数据库
================================================================================

代表: Cassandra, HBase, ClickHouse

核心优势:

    ✓ 列级压缩
      同类数据压缩效率高

    ✓ 写入吞吐高
      适合大规模数据写入

    ✓ 时序友好
      按时间分区查询

    ✓ 水平扩展
      分布式集群，PB 级存储

本质局限:

    ✗ 单点查询弱
      不适合精确找一条数据

    ✗ JOIN 弱
      不擅长关联查询

    ✗ 实时性差
      写入到可查有延迟

适合 Agent 记忆吗:

    适合: 海量历史数据归档、日志分析、长期统计
    不适合: 实时查询、小规模场景

================================================================================
六、时序数据库
================================================================================

代表: InfluxDB, TimescaleDB, Prometheus

核心优势:

    ✓ 时间索引原生
      按时间范围查询极快

    ✓ 压缩高效
      时序数据压缩率高

    ✓ 聚合友好
      时间窗口统计、趋势分析

    ✓ 写入吞吐高
      高频写入优化

本质局限:

    ✗ 只擅长时间维度查询

    ✗ 无复杂关系查询

    ✗ 无全文搜索

适合 Agent 记忆吗:

    适合: 时间轴分析（对话趋势、记忆热度变化）
    不适合: 内容检索、关系查询

典型用法:

    时序库存记忆访问频率，SQLite 存记忆内容
    用于实现"温度模型"（哪条记忆最近被访问）

================================================================================
七、对象存储
================================================================================

代表: AWS S3, 阿里云 OSS, MinIO（自建）

核心优势:

    ✓ 无限容量
      PB 级存储，无需担心扩容

    ✓ 成本极低
      归档存储成本几乎为零

    ✓ 高可用
      多副本、跨区域容灾

    ✓ 多版本
      支持版本管理

本质局限:

    ✗ 无查询能力
      只能按 key 取，不能搜索内容

    ✗ 延迟高
      网络 IO，不是本地磁盘

    ✗ 无结构化
      对象是黑盒文件

适合 Agent 记忆吗:

    适合: 长期冷数据归档、大文件（音频、图片）、备份
    不适合: 热数据、需要检索的数据

典型用法:

    SQLite 存索引和热数据，S3 存完整对话文件和附件
    实现"冷热分离"

================================================================================
八、内存数据库
================================================================================

代表: Redis（内存模式）、Memcached、VoltDB

核心优势:

    ✓ 微秒级响应
      内存操作，极致性能

    ✓ 无磁盘 IO
      响应时间稳定

    ✓ 适合临时数据
      缓存、会话、计数器

本质局限:

    ✗ 内存成本高
      GB 级内存 = 钱

    ✗ 持久化弱
      掉电丢失（虽然有 AOF/RDB，但不如磁盘可靠）

    ✗ 容量有限
      受内存大小限制

适合 Agent 记忆吗:

    适合: 缓存层、热数据加速、临时状态
    不适合: 长期存储、大容量数据

================================================================================
九、消息队列
================================================================================

代表: Kafka, RabbitMQ, Pulsar

核心优势:

    ✓ 流式处理
      事件驱动，实时管道

    ✓ 高吞吐
      百万级消息/秒

    ✓ 持久化
      消息可保留，支持回溯

    ✓ 解耦
      生产者消费者分离

本质局限:

    ✗ 不是存储
      是传输管道

    ✗ 查询弱
      只能顺序消费，不能随机查询

    ✗ 不是最终归宿
      消息最终要落库

适合 Agent 记忆吗:

    适合: 实时事件流（对话流入 → 处理 → 存库）
    不适合: 长期存储、查询检索

典型用法:

    Kafka 收对话事件流 → 实时处理提取关键信息 → 写入 SQLite
