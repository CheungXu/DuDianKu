================================================================================
二、关系数据库
================================================================================

代表: MySQL, PostgreSQL, SQLite

核心优势:

    ✓ 精确匹配
      WHERE 条件、JOIN 关联，毫秒级精确查询

    ✓ 结构化约束
      Schema 保证数据一致性，字段类型、外键约束

    ✓ 事务支持
      ACID 保证，多操作原子性

    ✓ 成熟生态
      SQL 标准、工具链完善、运维成熟

本质局限:

    ✗ 无法理解"相似"
      只能做 LIKE，不是语义相似

    ✗ 关系查询弱
      多表 JOIN 性能差，递归查询困难

    ✗ Schema 刚性
      改结构成本高

适合存什么:

    结构化数据、需要精确查询的数据、有固定 schema 的数据
    例: 用户信息、订单记录、标签表、元数据索引

不适合存什么:

    非结构化文本、需要"理解意思"的搜索、复杂关系网络

================================================================================
三、向量数据库
================================================================================

代表: Milvus, Qdrant, Pinecone

核心优势:

    ✓ 语义理解
      把"意思相近"量化为向量距离

    ✓ 模糊匹配
      不用精确关键词，也能找到相关内容

    ✓ ANN 搜索
      亿级向量，毫秒级相似度检索

    ✓ 多模态支持
      文本、图片、音频都能向量化

本质局限:

    ✗ 无法精确查询
      只能找"相似"，不能找"等于"

    ✗ 向量维度固定
      改模型要全量重建

    ✗ 不存结构化属性
      只能做辅助检索

    ✗ 解释性差
      为什么相似？很难解释

适合存什么:

    需要语义搜索的内容: 对话记忆、文档片段、知识条目
    例: 找"Python 异步方案"时，能匹配到"asyncio 教程"

不适合存什么:

    需要精确查询的数据、需要事务的数据、关系网络

================================================================================
四、文件系统
================================================================================

代表: 本地文件、Git

核心优势:

    ✓ 人可读
      Markdown、JSON、TXT 直接打开看

    ✓ 版本管理
      Git 友好，历史可追溯

    ✓ 可移植性
      拷贝文件夹就能迁移，无依赖

    ✓ 批量处理
      适合大数据导入导出

    ✓ 无 Schema
      灵活，随时改格式

本质局限:

    ✗ 查询能力弱
      只能 grep，不能 SQL

    ✗ 无索引
      全文搜索要额外工具

    ✗ 无事务
      并发写入容易冲突

    ✗ 性能差
      文件多了很慢

适合存什么:

    需要人查看的数据、需要版本管理的数据、备份归档
    例: 每日对话日志、知识笔记、配置文件、导出备份

不适合存什么:

    需要快速查询的数据、需要事务的数据、高频读写数据

================================================================================
五、图数据库
================================================================================

代表: Neo4j, Nebula, ArangoDB

核心优势:

    ✓ 关系原生
      节点-边模型，关系是第一公民

    ✓ 路径查询
      找"A 和 B 之间路径"、"所有关联节点"

    ✓ 递归遍历
      多层关系，一查询搞定

    ✓ 推理能力
      根据关系推断新知识

本质局限:

    ✗ 单点查询弱
      找一个节点不如关系库快

    ✗ 统计分析弱
      聚合、分组不如 SQL

    ✗ 学习成本高
      Cypher/Gremlin 语法特殊

    ✗ 存储成本高
      边越多，性能越差

适合存什么:

    关系密集的数据、需要路径查询的数据、知识网络
    例: 知识图谱、社交网络、组织架构、依赖关系

不适合存什么:

    独立的数据点、需要统计聚合的数据、简单查询场景
