一、核心理念
================================================================================

三层递进架构：

    牍（底层）→ 典（中层）→ 库（上层）

    ┌─────────────────────────────────────────────────────────────────────────┐
    │                                                                         │
    │   牍 - 最小记忆单元                                                     │
    │   职责: 全量存、可回溯、时间溯源                                        │
    │   内容: 原始对话、原始记录、带时间戳                                    │
    │   原则: 不筛选、不判断、全量保留                                        │
    │                                                                         │
    └─────────────────────────────────────────────────────────────────────────┘
                              ↑ 全量保留，不做筛选
    ┌─────────────────────────────────────────────────────────────────────────┐
    │                                                                         │
    │   典 - 凝练知识范式                                                     │
    │   职责: 快速检索、核心事实                                              │
    │   内容: 事件、标签、结论、精华                                          │
    │   原则: 只提取有价值的关键信息                                          │
    │                                                                         │
    └─────────────────────────────────────────────────────────────────────────┘
                              ↑ 只提取有价值的关键信息
    ┌─────────────────────────────────────────────────────────────────────────┐
    │                                                                         │
    │   库 - 体系化知识底座                                                   │
    │   职责: 知识导航、关系网络                                              │
    │   内容: 节点（概念）+ 边（关系）                                        │
    │   原则: 不存具体内容，存"节点+边"                                       │
    │                                                                         │
    └─────────────────────────────────────────────────────────────────────────┘

关键洞察：

    上层不是"压缩版底层"，而是"知识骨架"

    底层 = 档案室（全量存，用于回溯）
    中层 = 摘录本（精华存，用于检索）
    上层 = 百科索引（关联存，用于导航）

    三层是"不同用途"，不是"同一内容的三个版本"

--------------------------------------------------------------------------------
3.1 牍（底层）- 原始数据存储
--------------------------------------------------------------------------------

选型方案：

    ┌────────────────────┬────────────────────┬────────────────────┐
    │       类型         │      方案          │       理由         │
    ├────────────────────┼────────────────────┼────────────────────┤
    │     主存储         │   SQLite 单表      │ Python原生sqlite3  │
    │                    │                    │ 零依赖、单文件     │
    │                    │                    │ 实时写入、精确回溯 │
    ├────────────────────┼────────────────────┼────────────────────┤
    │     辅存储         │   Markdown 文件    │ 人可读、版本管理   │
    │                    │                    │ 灾难恢复、必须备份 │
    ├────────────────────┼────────────────────┼────────────────────┤
    │     可选层         │   DuckDB           │ 离线时间分析       │
    │                    │                    │ 聚合统计、趋势报告 │
    └────────────────────┴────────────────────┴────────────────────┘

查询需求分析：

    最频繁查询（实时）：
      - 精确回溯：找出某条对话 → SQLite（毫秒级）
      - 时间范围：找出某时间段对话 → SQLite（毫秒级）
      结论：SQLite 已足够快，无需 DuckDB 实时参与

    低频查询（离线）：
      - 大范围统计：最近一年每天对话数量 → DuckDB（定期批量）
      - 时间趋势：每小时对话热度 → DuckDB（定期批量）
      结论：DuckDB 作为离线分析工具，定期生成报告

    Markdown 备份（必须）：
      - 人可读、版本管理、灾难恢复
      - SQLite 崩溃时最后的防线
      - DuckDB 无法替代此功能

实现方式：

    实时写入：
      SQLite INSERT → 全量存入

    每日备份：
      SQLite → 导出 → Markdown 文件（按日期命名）
      例：backup/2026-03-31.md

    定期分析（可选）：
      SQLite → DuckDB 导入 → 时间分析 → 生成报告

向量方案对比：

    ┌────────────────┬──────────────┬──────────────┬──────────────┐
    │     方案       │   Python     │   嵌入式    │   云服务     │
    ├────────────────┼──────────────┼──────────────┼──────────────┤

      - pip install chromadb 即可
      - 嵌入式运行，无需Docker
      - API简洁，自动管理向量索引

核心查询能力：

    精确查询（SQLite）：
      SELECT * FROM dian_key WHERE tags LIKE '%async%'

    全文搜索（FTS5）：
      SELECT * FROM dian_key_fts WHERE dian_key_fts MATCH 'asyncio'

    语义搜索（Chroma）：
      collection.query(query_texts=["Python异步方案"], n_results=5)
      → 返回语义相似的关键信息

      nx.community.greedy_modularity_communities(G)
      → 发现知识群落

      nx.degree_centrality(G)
      → 找核心概念（连接最多的节点）

六、Phase 1 表结构设计
================================================================================

数据库：单 SQLite 文件（dudianku.db）

--------------------------------------------------------------------------------
表1：du_raw（牍 - 原始数据）
--------------------------------------------------------------------------------

    CREATE TABLE du_raw (
        id            TEXT PRIMARY KEY,
        timestamp     TEXT NOT NULL,
        type          TEXT,              -- conversation/log/event
        source        TEXT,              -- feishu/telegram/web
        raw_content   TEXT,              -- 原始数据 JSON
        metadata      TEXT               -- 元数据 JSON
    );

    CREATE INDEX idx_du_timestamp ON du_raw(timestamp);

--------------------------------------------------------------------------------
表2：dian_key（典 - 关键信息）
--------------------------------------------------------------------------------

    CREATE TABLE dian_key (
        id            TEXT PRIMARY KEY,
        source_du_id  TEXT,              -- 关联的牍ID
        type          TEXT,              -- knowledge/event/decision
        tags          TEXT,              -- JSON array
        event         TEXT,
        conclusion    TEXT,
        importance    REAL,              -- 0.0 ~ 1.0
        created_at    TEXT
    );

    CREATE INDEX idx_dian_type ON dian_key(type);
    CREATE INDEX idx_dian_importance ON dian_key(importance);

--------------------------------------------------------------------------------
表3：dian_key_fts（典 - 全文搜索索引）
--------------------------------------------------------------------------------

    CREATE VIRTUAL TABLE dian_key_fts USING FTS5(
        id,
        event,
        conclusion,
        tags,
        content='dian_key'
    );

--------------------------------------------------------------------------------
表4：ku_nodes（库 - 知识节点）
--------------------------------------------------------------------------------

    CREATE TABLE ku_nodes (
        id            TEXT PRIMARY KEY,
        name          TEXT NOT NULL,
        type          TEXT,              -- concept/entity/topic
        source_dian_ids TEXT,            -- JSON array
        properties    TEXT               -- JSON
    );

    CREATE INDEX idx_ku_name ON ku_nodes(name);

--------------------------------------------------------------------------------
表5：ku_edges（库 - 知识关系）
--------------------------------------------------------------------------------

    CREATE TABLE ku_edges (
        id            TEXT PRIMARY KEY,
        from_node     TEXT NOT NULL,
        to_node       TEXT NOT NULL,
        relation      TEXT,              -- "是方案"/"相关"/"依赖"
        weight        REAL,              -- 0.0 ~ 1.0
        source_dian_id TEXT
    );

    CREATE INDEX idx_ku_from ON ku_edges(from_node);
    CREATE INDEX idx_ku_to ON ku_edges(to_node);

七、数据流转示意
================================================================================

新对话进入流程：

    ┌────────────────┐
    │   对话输入     │
    └────────────────┘
           │
           ↓
    ┌────────────────┐
    │   牍表写入     │  ← 全量保留，INSERT du_raw
    │   (实时)       │
    └────────────────┘
           │
           ↓ 分析提取
    ┌────────────────┐
    │   典表写入     │  ← 判断是否关键信息
    │   (有价值)     │  ← INSERT dian_key + FTS5索引
    └────────────────┘
           │
           ↓ 关联分析
    ┌────────────────┐
    │   库表更新     │  ← 建立知识节点和边
    │   (关系)       │  ← INSERT/UPDATE ku_nodes/ku_edges
    └────────────────┘
           │
           ↓ 每日执行
    ┌────────────────┐
    │   Markdown备份 │  ← 导出每日对话
    │   (定时任务)   │
    └────────────────┘

查询流程：

    精确回溯（牍层）：
      SELECT * FROM du_raw WHERE timestamp BETWEEN ...

    内容检索（典层）：
      - 精确：SELECT * FROM dian_key WHERE tags LIKE ...
      - 全文：SELECT * FROM dian_key_fts WHERE ...
      - 语义：Chroma.query(query_texts=...)

    知识导航（库层）：
      - 查节点：SELECT * FROM ku_nodes WHERE name = ...
      - 查关系：SELECT * FROM ku_edges WHERE from_node = ...
      - 路径：NetworkX.shortest_path(G, ...)

八、选型理由总结
================================================================================

SQLite 全栈的理由：

    1. Python 原生 sqlite3，零安装
    2. 单文件，零配置，便携
    3. FTS5 内置，无需 Elasticsearch
    4. 三层同库，统一管理
    5. 100% 开源 MIT

Chroma（向量）的理由：

    1. 原生 Python，pip 一装即用
    2. 嵌入式运行，无需 Docker
    3. API 简洁，自动管理索引
    4. Apache 2.0 开源
    5. 云服务可选（Chroma Cloud）

NetworkX（图算法）的理由：

    1. Python 原生图算法库
    2. 路径查找、社区检测、关系分析
    3. BSD 开源
    4. 轻量，无外部依赖

Markdown 备份的理由：

    1. 人可读，直接打开查看
    2. Git 版本管理友好
    3. 灾难恢复，SQLite崩溃时最后防线
    4. Python 原生，零依赖
    5. 不可替代的备份价值

DuckDB（可选）的理由：

    1. 离线时间分析，聚合统计
    2. Python 原生，pip 安装
    3. 单文件嵌入式
    4. Apache 2.0 开源
    5. 低频需求，定期批量分析即可
