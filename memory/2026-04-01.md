# 2026-04-01 DuDianKu 开发日志

## 典层触发机制设计（三阶段）

### 时间：00:31

与张旭讨论确定典层触发机制的三阶段设计：

---

### 三阶段概览

| 阶段 | 触发类型 | 输出形态 | 定位 |
|------|----------|----------|------|
| Phase 1 | 准实时触发 | 临时典 | 快速响应 |
| Phase 2 | 定时+条件触发 | 正式典 | 批量整合 |
| Phase 3 | 热度触发 | 紧急典 | 补充机制 |

---

### Phase 1: 准实时触发

- **触发时机**: 对话间隙（无Tool调用的回复后）
- **输出**: 临时典（is_temporary = true）
- **生命周期**: 暂存典层，等待定时整合决定升格或淘汰
- **优势**: 有热度属性，可追踪访问频率

### Phase 2: 定时+条件触发

- **触发条件**: 三重门控
  - 时间门：>= 24h
  - 数据门：>= 50条新牍
  - 锁门：无其他进程整合中
- **输出**: 正式典（进入库层索引）
- **执行**: 后台异步，ForkedAgent模式
- **整合流程**: 扫描→筛选→建立关联→更新索引→清理锁

### Phase 3: 热度触发

- **触发条件**: 牍heat_score >= 80
- **输出**: 紧急典（trigger_type = 'heat_threshold'）
- **冷却期**: >= 4小时防频繁触发
- **独有优势**: CC没有热度概念，这是差异化设计

---

### 实现优先级

```
Phase 1（最高）: 准实时触发
  - 临时典表/字段
  - 对话间隙识别
  - 准实时触发器

Phase 2（核心）: 定时+条件触发
  - 三重门控机制
  - ForkedAgent异步整合
  - 去重合并算法
  - 关联关系建立

Phase 3（补充）: 热度触发
  - 热度计算公式完善
  - 热度阈值触发器
  - 冷却期机制
```
